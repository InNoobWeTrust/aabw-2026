## Naming Drifts Resolved

This glossary codifies three specific migrations from the current codebase to the canonical domain language:

### 1. `pending` ‚Üí `queued`
- **Where it exists:** `backend/models.py:25` `JobStatus.PENDING = "pending"`; `backend/routes.py:205` `status=JobStatus.PENDING`; `frontend/app.js:5` `"pending"` in ACTIVE_STATUSES; `frontend/app.js:344` label `"Pending"`.
- **Why change:** "pending" is ambiguous ‚Äî it suggests the system hasn't acknowledged the request. "queued" correctly describes a Job that has been accepted, assigned an ID, and is waiting for a pipeline worker slot.
- **Migration:** add `QUEUED = "queued"` to JobStatus, deprecate `PENDING`, update all references. The old value `"pending"` becomes a prohibited alias.

### 2. Stage names leaked into `JobStatus` enum
- **Where it exists:** `backend/models.py:26-31` ‚Äî `PREPROCESSING`, `POSE_ESTIMATION`, `RETARGETING`, `EVALUATING`, `PACKAGING` are both status enum members AND used as `current_stage` strings in `backend/routes.py:60-129`.
- **Why change:** Status and stage are orthogonal concepts. A Job's status is "processing" (or "running") while it moves through multiple stages. Embedding stage names into the status enum forces false states (e.g., a Job cannot be "preprocessing" AND "completed" simultaneously, but mixing the enums makes the type system allow nonsensical combinations). The canonical stage names are: `ingest`, `preprocess`, `pose`, `retarget`, `evaluate`, `package`, `finalize`.
- **Migration:** separate `JobStatus` into a compact lifecycle enum (`queued`, `processing`, `completed`, `failed`) and introduce a `PipelineStage` enum. The `current_stage` field keeps its purpose but references the stage enum, not the status enum.

### 3. Global admin subject ‚Üí `judge_session_id + AdminRole`
- **Where it exists:** `backend/routes.py:157` `create_access_token({"sub": "admin"})` ‚Äî a bare `"admin"` string with no session context; `backend/auth.py:28` `get_current_user` returns the raw payload.
- **Why change:** A single global admin token means all actions are unattributable. When the system grows to support multiple judges, contests, or audit requirements, every action must be traceable to a specific judge session. The canonical identity model is `judge_session_id` (which contest/event) + `AdminRole` (what privileges within that session).
- **Migration:** extend the JWT payload to include `judge_session_id` and a scoped role claim. Deprecate the bare `"sub": "admin"` pattern.
